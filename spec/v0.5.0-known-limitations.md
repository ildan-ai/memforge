# MemForge v0.5.0 known limitations

**Date:** 2026-05-10
**Status:** Published with v0.5.0 spec.

---

## Why this document exists

This document tracks residual non-BLOCKER refinements queued for v0.5.x patches. The two BLOCKER-class issues originally identified during v0.5 development have been **closed in v0.5.0 normative spec**:

- **Receiver-state silent-rollback window** → closed via §"Receiver state (MUST)" mandating the receiver-state file shape (`<memory-root>/.memforge/receiver-state/<sender-uid>.yaml`; FS mode 0600 + 0700; reject `seq <= highest_seen_sequence`; HALT on corruption).
- **Remote-fetch unsigned revocation events** → closed via §"Revocation events as git commits" mandating GPG signature on every revocation commit + §"Sparse-checkout / shallow-clone fallback verification mode" mandating TOFU + fast-forward-only + signature verification on fetched commits.

v0.5.0 ships with no BLOCKER-class known limitations.

---

## Residual MAJORs (v0.5.x patch targets)

These are refinements to in-spec mechanisms; not security gaps. Tracked for v0.5.1 / v0.5.x release windows.

### MAJOR 1: Element 4 signed checkpoint signer ambiguity for agent ephemeral keys

The signed checkpoint mechanism specifies "signed by operator's long-lived key" but agents have ephemeral per-session keys. v0.5.0 spec language treats checkpoint signer as the sender's writing key (agent ephemeral OR operator long-lived) with operator-uuid bound in scope, but the language could be tightened. v0.5.x defines the canonical checkpoint-signing flow.

### MAJOR 2: Element 5 revocation snapshot ancestor verification

`memforge: revocation-snapshot <hash>` commits MUST be ancestor of the verified head; spec needs explicit fast-forward-only check on snapshot commits + HALT on competing snapshot lines. Implicit in current language; explicit normative statement deferred to v0.5.1.

### MAJOR 3: Element 5 revocation snapshot canonical hash

Canonical ordering of revoked key_ids + exact hash algorithm + encoding undefined; spec must define for v0.5.1.

### MAJOR 4: Sender / receiver posture nuance

Sender-side auto-new-uid on file-corruption is intentional rotation; receiver-side HALT on corruption preserves replay-protection floor. The asymmetry is documented in §"Receiver state" and §"Sender-sequence file deletion / corruption recovery" but the cross-cutting design rationale could be more prominent. v0.5.x adds a consolidated §"Recovery posture across mutable internal files" subsection.

### MAJOR 5: Cache TTL semantics for high-stakes deployments

The default 1-hour TTL for revocation cache balances freshness vs network load. High-stakes deployments may want TTL=0 (fetch on every adapter startup); spec needs explicit documentation that TTL contributes to the vulnerability window between key revocation and propagation.

### MAJOR 6: Agent session attestation content scope

Agent session attestations are signed by the operator's long-lived key, but the spec does not constrain the attestation's content. v0.5.1 defines the attestation envelope: nonce + expiry (≤ 24 hours; configurable) + capability scope (read-only / write-capable / specific decision-topic limit / etc.). Closes the agent-attestation-spoofing surface.

## Residual MINORs (v0.5.x patch targets)

### MINOR 1: Cross-cutting fail-closed posture documentation

Document fail-closed/recovery semantics for ALL mutable MemForge-internal files (operator-registry HALT; revocation cache rebuild-or-HALT; sender-sequence new-sender-uid; receiver-state HALT; checkpoint cache HALT) + specify startup verification order. v0.5.x consolidation pass.

### MINOR 2: Cache TTL semantics in remote-fetch fallback

Document TTL semantics + edge cases for revocation cache in remote-fetch fallback mode. Includes what happens on TTL=0, on TTL > snooze-horizon, etc.

### MINOR 3: Trust graph disclosure documentation

The `operator-registry.yaml` file discloses the project's trust graph (operators, key rotation history, join dates). This is by design for transparency + auditability, but operators in adversarial environments may want to know the trade-off. v0.5.x adds a "Privacy considerations" subsection to the spec.

### MINOR 4: Agent session ID format guidance

The example `agent-session-id` (`cc-2026-05-10-aaaa`) leaks toolchain + date + session metadata. v0.5.0 does not constrain the format; v0.5.x recommends opaque random identifiers to avoid metadata leakage to anyone with read access.

### MINOR 5: Unbounded key rotation chain DoS guidance

Spec recommends fresh-start every 10 rotations (operator discretion). v0.5.x adds adapter-side `max_rotation_chain_length` config (default 20) with persistent MAJOR audit warning when exceeded.

### MINOR 6: Operator name homograph audit

`memory-audit` SHOULD warn on new operator-registry additions whose `operator_name` has Levenshtein distance ≤ 2 from any existing operator's name. Defends against visually-similar Unicode substitutions in trust-bootstrap step. v0.5.x patch.

### MINOR 7: v0.4 memory flooding audit MAJOR rate-limit

Audit emits a one-time MAJOR per unsigned v0.4 memory under v0.5 readers. An attacker with write access could flood the repository with thousands of valid v0.4 memories, generating audit noise. v0.5.x adds `audit.v04_unsigned_memories_rate_limit` config to cap the count of such MAJORs reported per audit run (default 10; rest are summarized).

## Reference CLI + operator-facing documentation gaps (ship as v0.5.1 patches)

- `memforge init-operator` — interactive operator-identity + recovery-secret bootstrap.
- `memforge init-store` — initial 1-operator self-signed registry creation in a target repo.
- `memforge operator-registry add | verify | remove | fresh-start` — registry management.
- `memforge revoke <key_id>` + `memforge revocation-snapshot` + `memforge memories-by-key <key_id>` + `memforge revoke-memories --bulk` — revocation flow CLI.
- `memforge rotate-key` — operator-key rotation CLI.
- `memforge upgrade-v04-memories` — bulk re-sign of v0.4 memories under current operator identity.
- `memforge migrate-claim-block` — already shipping in v0.4.x.
- `memforge revoke-cache-refresh` — manual cache bust for remote-fetch fallback mode.
- `memforge messaging-doctor` — adapter health check (sender-sequence + checkpoint + receiver-state validation).
- `memforge recovery-init --hardware <yubikey|secure-enclave|tpm>` — hardware-backed recovery-secret install (v0.5.x).

The CLI commands describe the v0.5.0 contract; binary delivery is v0.5.1 scope.
